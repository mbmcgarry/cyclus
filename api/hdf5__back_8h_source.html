00017 /// An Recorder backend that writes data to an hdf5 file.  Identically named
00018 /// Datum objects have their data placed as rows in a single table.
00019 ///
00020 /// The HDF5 backend ensures that every column in its tables is represented
00021 /// in the schema with a fixed size.  This in turn ensures that the schema itself
00022 /// is of a fixed size. This fixed size constraint applies even to variable length
00023 /// (VL) data types (string, blob, vector, etc).
00024 ///
00025 /// Variable length data is handled in a special way to ensure a fixed length
00026 /// column.  The naive approach would be to set a maximum size based on the data
00027 /// available. However, this is not truly a fixed length data type. Instead, the
00028 /// HDF5 backend serves as an on-disk bidriectional hash map for each VL data type.
00029 ///
00030 /// A regular hash table applies a hash function to keys and stores the values based
00031 /// on this hash. Keys are unique and values may be repeated for many keys. In a
00032 /// bidirectional hash map the keys and values are both one-to-one and onto. This
00033 /// makes storing a seperate hash redunant and since the key and hash are the same.
00034 ///
00035 /// The HDF5 backend uses the well-known SHA1 hash function as its keys for VL data.
00036 /// This is because SHA1 is 5x the size (20 bytes) of standard 32-bit unsigned ints.
00037 /// This provides a gigantic address space in which to store variable length data.
00038 /// HDF5 provides two critical features that make an address space of this size
00039 /// possible: multidiemnsional arrays and chunking.
00040 ///
00041 /// HDF5 easily supports 5D arrays. This allows us to use the SHA1 hash not only as
00042 /// a key, but also a we can cast it to an index (len-5 array of unsigned ints) for
00043 /// a 5D array.  Furthermore, for such an array we can set the chunksize to a single
00044 /// element ([1, 1, 1, 1, 1]). This allows us to have the full space available
00045 /// ([UINT_MAX, UINT_MAX, UINT_MAX, UINT_MAX, UINT_MAX]) while only storing the
00046 /// data that actually exists!
00047 ///
00048 /// In the table columns for VL data, the HDF5 backend stores the SHA1 as a length-5
00049 /// array of unsigned ints. Looking up the associated value is simply a matter of using
00050 /// this array as an index into a special data type array as above.
00051 /// This has the added advantage of de-duplicating storage for identical entries.
00052 ///
00053 /// On disk the keys and values for a data type are stored as arrays named with
00054 /// base data type and the string "Keys" and "Vals" appended respectively.  For
00055 /// instance, BLOB is stored in the arrays BlobKeys and BlobVals while VL_VECTOR_INT
00056 /// is stored in the arrays VectorIntKeys and VectorIntVals.
00057 ///
00058 /// In memory, all active keys are stored in vlkeys_ private member of this class.
00059 /// This maps the DbType to a set of the SHA1 digests. This is used to prevent
00060 /// excessive writing of values to disk that already exist.
00061 ///
00062 /// The cost of the bidirectional hash map strategy is that the values need to be
00063 /// looked up in a separate read() from that of the table itself.  However, by
00064 /// using VL data types users should expect a performance hit and this is one of
00065 /// the more effiecient strategies.
00066 ///
00067 /// Another implicit problem with all hash mappings is the possibility of collision.
00068 /// However, this is in practice impossible here.  For SHA1, there is a 3.4e-13 chance
00069 /// of having a single collission with 1e18 (a billion billion) entries.
00070 ///
00071 /// Still, if the address space of SHA1 ever becomes insufficient for some reason,
00072 /// please  move to a larger SHA value such as SHA224 or SHA256 or higher. Such a
00073 /// migration is not anticipated but would be straighforward.
