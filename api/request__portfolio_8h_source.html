00053 /// @class RequestPortfolio
00054 ///
00055 /// @brief A RequestPortfolio is a group of (possibly constrainted) requests for
00056 /// resources
00057 ///
00058 /// The portfolio contains a grouping of resource requests that may be mutually
00059 /// met by suppliers. These requests may share a common set of
00060 /// constraints. Take, for instance, a facility that needs fuel, of which there are
00061 /// two commodity types, fuelA and fuelB. If some combination of the two suffice
00062 /// the facility's needs, then requests for both would be added to the portfolio
00063 /// along with a capacity constraint.
00064 ///
00065 /// The ExchangeTranslator will automatically add a mass-based constraint, that
00066 /// accounts for mutual requests, if the portfolio has them. , e.g.,
00067 /// @code
00068 ///
00069 /// RequestPortfolio<SomeResource>::Ptr rp(new RequestPortfolio<SomeResource>());
00070 /// // add some requests
00071 /// rp->AddRequest(/* args */);
00072 /// // declare some of them as multicommodity requsts (i.e., any one will
00073 /// // satisfy this demand).
00074 /// rp->AddMutualReqs(/* args */);
00075 ///
00076 /// @endcode
00077 ///
00078 /// A default constraint will add unity for normal requests in the portfolio,
00079 /// but will add a weighted coefficient for requests that meet the same mutual
00080 /// demand. For example, if 10 kg of MOX and 9 kg of UOX meet the same demand
00081 /// for fuel, coefficients are added such that a full order of either will
00082 /// determine the demand as "met". In this case, the total demand is 9.5, the
00083 /// MOX order is given a coefficient of 9.5 / 10, and the UOX order is given a
00084 /// coefficient of 9.5 / 9.
